# Career Guide

这份文档用于求职收尾，包含可直接复制的简历描述、面试表达和演示建议。

## 简历项目描述

### 极简版

基于 LangGraph 和 FastAPI 实现中文客服 Agent 系统，支持条件路由、多 Agent 协作、HITL 审批、混合检索 RAG 与持久记忆。

### 标准版

基于 LangGraph、LangChain 和 FastAPI 搭建中文客服 Agent Demo，围绕知识问答、业务查询、人工升级等场景实现条件路由、多 Agent 协作、HITL 审批、混合检索 RAG 与线程级恢复，形成可通过 Swagger 直接演示的完整后端项目。

### 简历三点版

- 使用 LangGraph 设计客服多 Agent 工作流，按意图、风险和上下文在知识检索、工具执行、人工升级之间条件路由。
- 实现 middleware、HITL 审批、线程级状态恢复和用户级长期记忆，提升高风险动作的可控性和会话连续性。
- 基于 FastAPI 提供 REST、SSE、WebSocket 与测试闭环，支持 Swagger 演示、RAG 调试和真实业务工具调用。

## 推荐写法

如果简历空间很小，用“极简版”。

如果项目区能写两到三行，用“标准版 + 简历三点版”。

如果面试官更看重工程能力，优先强调下面几个词：

- LangGraph 条件路由
- 多 Agent 协作
- HITL 审批与恢复
- 混合检索 RAG
- 持久记忆
- FastAPI 工程化落地

## 30 秒面试介绍

这是一个中文客服 Agent Demo。我没有把它做成单条 chain，而是用 LangGraph 搭了条件路由图，把知识问答、业务执行、人工升级拆成不同路径；同时加了 middleware、HITL 审批、混合检索 RAG 和长短期记忆，最后通过 FastAPI 和 Swagger 把它做成一个可以直接演示和测试的工程化项目。

## 90 秒面试介绍

这个项目的核心不是“接一个大模型 API”，而是把 Agent 真正做成一个后端系统。架构上我用 LangGraph 做条件边路由，先分析用户请求，再根据意图、风险和上下文进入知识检索、业务工具或人工升级路径。高风险动作不会直接执行，而是先触发 HITL 中断，审批后再恢复同一线程继续执行。

在 RAG 部分，我没有只做单一向量检索，而是做了混合检索和轻量增强，包含查询规范化、子问题拆分、融合与重排，让返回结果更稳定。记忆部分区分了线程级短期状态和用户级长期记忆，既能保留对话上下文，也能在跨轮次场景下召回用户事实和偏好。

工程侧我用 FastAPI 提供 REST、SSE、WebSocket 和 Swagger，用测试覆盖核心流程，所以它不是概念验证，而是一个可以运行、可以演示、可以继续扩展的 Agent 项目。

## 面试官可能会追问什么

### 为什么用 LangGraph，而不是普通 Agent 或 chain

因为这个项目里存在明确的条件分支、高风险动作审批、可恢复执行和多路径协作，图结构比单链路更适合表达真实业务流程，也更方便做状态管理和调试。

### 你的多 Agent 体现在哪里

不是简单把所有提示词堆在一个节点里，而是把知识检索、动作执行、人工升级等职责拆开，由分析节点和条件边决定走哪条路径，再经过统一校验和回复节点收口。

### 你说的 RAG 升级体现在哪里

重点不是堆复杂组件，而是做有效增强：混合检索、查询规范化、子问题拆分、融合与重排，再把引用和调试信息带回响应，保证可追踪性和准确率。

### 记忆是怎么做的

线程级状态用于恢复当前对话执行过程，用户级长期记忆用于存储事实、偏好和历史问题，下一轮可按语义相关性召回，解决跨轮次一致性问题。

## 演示时建议强调

- 先演示 FAQ，让面试官看到知识检索路径
- 再演示套餐或账单查询，让面试官看到工具调用路径
- 再演示创建工单或转人工，让面试官看到 HITL 中断与恢复
- 最后说明线程状态、长期记忆、Swagger 和测试，强调工程完整度

## 不建议的表达

- 不要只说“我做了一个智能客服”
- 不要只强调模型选型，而忽略图结构和系统设计
- 不要把重点放在“调用了多少工具”，而要放在“为什么这样编排更合理”
