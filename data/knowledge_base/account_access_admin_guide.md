# 账户访问与安全管理指南

本文档用于覆盖登录恢复、注册邮箱不可用、2FA、权限变更和所有权转移等高风险场景，适合作为账户安全问题的统一说明口径。对这类问题，客服应优先确认身份、权限和组织关系，再决定是直接提供操作指引，还是要求补充材料并进入人工审核。

## 适用范围

本指南面向管理员、工作区负责人、账单联系人以及需要处理高风险账户问题的支持人员。若用户只是普通登录失败且仍可访问注册邮箱，通常只需提供标准步骤；若用户无法访问注册邮箱、请求关闭 2FA、要求提升角色或转移所有权，则需要补充更多背景信息，并根据组织策略决定是否进入人工审核。

## 登录恢复

### 标准密码重置流程

用户仍能访问注册邮箱时，应优先走标准密码重置流程：进入“设置 > 安全中心 > 密码”，发起重置请求，查收系统邮件，在 24 小时内完成链接验证，并设置新密码。客服回复时需要同步说明，旧链接在重新发起后会立即失效，多个浏览器窗口同时打开旧邮件可能导致“链接无效”，企业邮箱安全网关也可能改写链接导致失败。若用户继续追问“为什么链接失效”“为什么收不到邮件”，应引导其按顺序排查邮箱地址是否正确、邮件是否被公司安全策略隔离、浏览器缓存是否导致旧链接重复打开，以及账号状态是否存在异常。

若问题仍未解决，客服还可以提醒用户先尝试退出所有设备、重新触发重置，并确认是否最近修改过主邮箱或安全设置。这样既能帮助用户先完成自助排查，也能减少不必要的人工恢复请求。

### 注册邮箱不可用时的身份核验

如果用户已无法访问注册邮箱，就不能直接发送新密码或直接替换邮箱，因为这属于高风险身份恢复。标准要求是先创建身份核验记录，并收集组织名称、历史付款记录、常用登录地点、最近一次成功登录时间、可验证的企业邮箱和紧急联系人信息。对于 Enterprise 组织，还建议补充合同编号、采购联系人或管理员名单。只有在人工审核通过后，后续恢复动作才可继续执行，例如补发重置链接、修改通知邮箱或安排管理员协助恢复。

在面向用户的答复里，建议明确说明审核的目的不是增加阻力，而是为了防止他人冒充账户所有者。若组织存在多管理员机制，还可以提醒用户同步联系其他管理员确认账号归属和恢复需求，以便更快完成核验。

## 双重身份验证与恢复码

### 2FA 开启、换机与停用

2FA 默认支持验证码 App 和短信两种方式，但高权限账号更推荐使用验证码 App。若用户只是更换手机且仍能登录，应先使用旧设备或恢复码完成迁移，再删除旧设备；若用户已经丢失设备且没有恢复码，系统不能直接停用 2FA，而应先走身份核验并由管理员或支持团队确认。

对于企业组织，还建议补充说明：2FA 停用不会自动撤销 SSO、IP 白名单或审计策略，管理员在恢复账号访问后仍应检查其他安全控制是否保持有效。若用户反馈验证码 App 和短信都不可用，回复中还应说明当前是否仍有已登录设备、是否可以使用恢复码，以及是否需要管理员辅助确认。

### 恢复码管理建议

每次开启 2FA 后，系统都会生成一组恢复码。帮助中心建议管理员将恢复码保存在企业密码管理器或合规的安全文档中，而不是存放在聊天工具或公开共享文档里。若用户询问“恢复码有什么用”，标准解释应为：恢复码用于设备丢失、验证码 App 无法访问或跨设备迁移时的紧急恢复，不应作为日常登录方式。

如果用户已经用掉了部分恢复码，建议提醒其在下一次成功登录后重新生成并替换旧码。对于团队管理员，还可以补充“恢复码应与离职流程、设备回收和管理员交接一起检查”的建议，避免安全文档长期失效。在更正式的安全管理场景下，也建议组织定期检查恢复码保存策略，确认只有需要的人能访问这些敏感信息。

## 权限与角色变更

### 角色提升的审批边界

平台默认提供管理员、编辑者和访客三类角色。角色提升通常意味着更高的配置能力、成员管理能力或账单可见性，因此不应通过普通聊天直接执行。标准流程是先核对发起人身份、确认目标成员当前角色、说明提升后的权限范围，再根据组织策略决定是否需要人工审核。涉及批量提权、账单联系人变更、API Key 管理权限开放或删除安全限制的请求，应默认进入审批。

如果用户只是问“管理员和编辑者有什么区别”，系统可以直接给出说明型答复；如果用户说“把我同事直接提成管理员”，则应先记录谁发起、要改谁、改完后会获得哪些关键权限。若目标成员尚未接受邀请或还未完成首次登录，也应提醒对方先激活账号，再继续处理权限调整。

### 账户所有权转移

所有权转移不仅会改变管理权限，还可能影响账单联系人、合同续签、数据导出和安全通知的接收对象。标准流程要求当前负责人和目标负责人都完成确认；若原负责人离职或无法联系，应补充组织级证明材料并进入人工审核。客服在回复时应说明：只有审核通过后，所有权变更才会继续执行，且建议在变更前先确认未完成工单、账单联系人、API Key 和安全告警联系人是否需要同步调整。

对于大型组织，还建议在回复里提醒用户提前评估交接窗口，例如是否避开月底结算、是否需要先轮换 API Key、是否有正在处理的安全告警或人工工单。若用户只是想临时授权他人代管，而不是永久转移所有权，则更适合改走角色提升或协作管理路径，而不是直接变更 owner。

### 审计记录与通知

对于密码恢复、2FA 停用、角色提升和所有权转移等高风险操作，系统应记录发起时间、发起人、目标对象、审批状态、审批原因和恢复后的执行结果。建议在用户可见回复中也保留一层轻量通知，例如“本次请求需要人工审核，审核完成后会通过站内通知和邮件同步结果”。这样既能帮助用户管理预期，也便于组织内部留存可追溯的处理记录。

对于管理员较多的组织，还可以提醒在通知里同步抄送相关责任人，避免权限或安全变更只被单个人看到。对于涉及多个部门的变更，也建议在说明中标注预计处理窗口和当前责任方。
